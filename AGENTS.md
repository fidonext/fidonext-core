# Project: FidoNext Core

## Description
- Low-level networking and E2EE core for the FidoNext decentralized communication ecosystem
- Decentralized P2P mesh (no central servers), censorship-resistant, privacy-first — inspired by the original FidoNET
- Ships a Rust library with a C-ABI surface so other languages/platforms (Python, C/C++, Android via JNI) can embed the node

## Stack
- **Language**: Rust 2021 edition, toolchain 1.91.0
- **Package manager**: Cargo
- **Async runtime**: Tokio 1.37 (multi-threaded)
- **Networking**: libp2p 0.56 — Gossipsub, Kademlia DHT, QUIC, TCP, WebSocket, Relay (client + server), Rendezvous, AutoNAT, Identify, Ping
- **Cryptography**: libsignal (X3DH + Double Ratchet), x25519-dalek, ChaCha20-Poly1305, SHA2, HKDF
- **Serialization**: serde, serde_json, CBOR
- **Storage**: sled (embedded KV for offline mailbox & profile state)
- **FFI**: cbindgen (generates `cabi-rust-libp2p.h` from Rust source at build time)
- **Logging**: tracing + tracing-subscriber (opt-in via `config::init_tracing()`)

## Architecture
- Main crate lives in `c-abi-libp2p/` (~13K LOC). The repo root is mostly docs/CI.
- **Command-driven event loop**: `PeerManager::run()` is a single async task using `tokio::select!` over a command mpsc channel + libp2p `SwarmEvent`s. Commands: `StartListening`, `Dial`, `FindPeer`, `PublishMessage`, `ReserveRelay`, etc.
- **Queue-based FFI (no callbacks)**: FFI clients *poll* dequeue functions — `cabi_node_dequeue_message`, `cabi_node_dequeue_discovery_event`, `cabi_node_dequeue_addr_event`, `cabi_node_dequeue_file_transfer`. Queues have fixed capacities (msg 256, discovery 512, file 256).
- **Composite libp2p NetworkBehaviour**: Kademlia + Ping + Identify + AutoNAT + Gossipsub + RelayClient/Server + Rendezvous + custom `DeliveryDirect` + `FileTransfer`.
- **Profile-based identity**: JSON profile file stores `account_seed`, `device_id`, `libp2p_seed`, `signal_identity_seed` and libsignal state. Same profile → same `PeerId`.
- **Offline mailbox**: sled-backed store for undelivered messages (up to 256 per recipient / 512 KB total). Fetched by recipient every ~8s with jitter.
- **Strict libsignal mode**: auto E2EE enforces official prekey bundle format; legacy explicit prekey paths are disabled.
- **Dual crate output**: cdylib (`.so`/`.dylib`/`.dll`) for FFI + rlib for Rust consumers. Header generated by `build.rs`.

## Key Files
- `c-abi-libp2p/src/lib.rs` — C-ABI surface (~2.5K lines, 50+ `#[no_mangle] pub extern "C"` functions). Also re-exports the Rust API.
- `c-abi-libp2p/src/peer/manager.rs` — `PeerManager` event loop (owns the Swarm, ~2.4K lines)
- `c-abi-libp2p/src/transport/libp2p.rs` — transport + behaviour + protocol config (`TransportConfig`)
- `c-abi-libp2p/src/peer/discovery.rs` — Kademlia discovery events
- `c-abi-libp2p/src/peer/addr_events.rs` — address/listen/AutoNAT/relay-ready events
- `c-abi-libp2p/src/peer/mailbox_store.rs` — sled-backed offline mailbox
- `c-abi-libp2p/src/messaging/{messaging,delivery,file_transfer}.rs` — message queue, delivery envelopes/ACK/NACK, chunked file transfer
- `c-abi-libp2p/src/e2ee/mod.rs` — libsignal integration (~3.7K lines): prekey bundles, key updates, encryption
- `c-abi-libp2p/src/config.rs` — global config & tracing init
- `c-abi-libp2p/build.rs` — cbindgen header generation
- `c-abi-libp2p/cabi-rust-libp2p.h` — generated C header (checked in)
- `c-abi-libp2p/Cargo.toml` — deps + `[lib] crate-type = ["cdylib", "rlib"]`
- `c-abi-libp2p/examples/rust/e2ee_local_mesh.rs` — Rust 2-node E2EE smoke test
- `c-abi-libp2p/examples/python/` — ctypes-based Python clients and `run_local_mesh.sh` harness
- `c-abi-libp2p/docs/networking_overview.md`, `relay-policy-offline-mailbox-spec.md`, `node-capabilities-policy.md`
- `.github/workflows/libcabi-build-and-release.yml` — multi-platform CI
- `Dockerfile` — multi-stage (builder / test / artifacts) for Linux x86_64, aarch64, Windows GNU, Android arm64-v8a
- `alice.profile.json` — sample E2EE profile

## Conventions
- Run `cargo fmt` before committing — uses default `rustfmt` (no custom `rustfmt.toml`)
- Run `cargo clippy` — no special allow/deny configuration
- Doc comments: `//!` at module top, `///` on public items
- Errors: internal code returns `anyhow::Result<T>`; FFI boundary returns `c_int` status codes (`CABI_STATUS_SUCCESS`, `CABI_STATUS_INVALID_ARGUMENT`, …)
- All FFI entry points are `#[no_mangle] pub extern "C" fn cabi_*` — keep the `cabi_` prefix
- Naming: `UPPER_SNAKE_CASE` constants, `PascalCase` types, `snake_case` functions
- Tests are inline (`#[test]` / `#[tokio::test]`) in source files — no top-level `tests/` directory

## Commands
Run from `c-abi-libp2p/` unless stated otherwise.

Build:
- `cargo build` — debug cdylib + rlib
- `cargo build --release`
- `cargo build --target aarch64-unknown-linux-gnu` — cross-compile (install target first)
- `cargo ndk -t arm64-v8a build --release` — Android build (for `fidonext_android`)

Run examples:
- `cargo run --example e2ee_local_mesh` — Rust 2-node E2EE mesh
- `python3 examples/python/ping_two_nodes.py` — ctypes ping test
- `python3 examples/python/ping_two_nodes.py --use-quic`
- `E2EE_MODE=on bash examples/python/run_local_mesh.sh` — full mesh harness (or `off` for cleartext)

Test & lint:
- `cargo test --release --verbose`
- `cargo clippy`
- `cargo fmt`

Docker (from repo root):
- `docker build -t fidonext-libp2p .` — all platforms
- `docker build -t fidonext-libp2p --target test .` — test-only

## Environment Variables
- `RUST_LOG` — tracing verbosity. Examples: `info`, `info,peer=debug,ffi=debug`
- `E2EE_MODE` — `on`/`off` in Python example harness
- `FIDONEXT_C_ABI` — path to compiled `.so`/`.dll` for Python examples (default: `target/debug/libcabi_rust_libp2p.so`)
- `FIDONEXT_FORCE_STDIN` — force stdin polling in Python examples (CI)
- `MESSAGE_DELAY`, `POST_MESSAGE_WAIT` — Python harness timing

## Runtime Configuration
Built via Rust API (also exposed through C-ABI):
```rust
TransportConfig::new(use_quic: bool, relay_server: bool)
    .with_identity_seed([u8; 32])         // deterministic keypair
    .with_bootstrap_peers(vec![Multiaddr]) // initial DHT peers
```
Profiles loaded/created via `e2ee::load_or_create_profile(path)`.

## Known Issues / Non-Obvious
- **Polling-only FFI**: consumers must loop on `cabi_node_dequeue_*`; messages are NOT pushed. Queue overflow drops silently — poll often enough.
- **Tracing is opt-in**: call `config::init_tracing()` from your binary/test. Safe to call multiple times (`OnceCell`).
- **Profile reuse = identity reuse**: reusing the same profile file yields the same `PeerId` and libsignal state across restarts. Delete the file to rotate identity.
- **Offline mailbox fetch cadence**: ~8s + jitter; expect small latency when a recipient reconnects.
- **Strict libsignal only**: don't reintroduce the old explicit prekey/session auto paths — they are intentionally disabled.
- **Test coverage is thin**: only ~2 inline tests (e2ee smoke, file transfer roundtrip). Main validation is the example mesh runs + CI smoke tests.
- **cbindgen regenerates the header at build time**: if you add a new `#[no_mangle]` FFI function, rebuild to refresh `cabi-rust-libp2p.h`, then commit it.
- **Cross-compile toolchains**: Linux aarch64 / Windows GNU / Android require the right Rust targets and linkers (see `.github/workflows/libcabi-build-and-release.yml` for the exact setup used in CI).
- **License**: AGPL-3.0 — keep license headers intact; derivative network services must disclose source.
